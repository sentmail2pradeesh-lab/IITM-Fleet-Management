1. LDAP integration (requirement)

The current production login path uses email and password stored in the application database (POST /api/auth/login with bcrypt verification). For institute single sign-on, the following is required:

  • Directory: Bind to IIT Madras LDAP (host, base DN, service account or anonymous bind as per campus policy) and authenticate the user with the submitted credentials, or validate a campus SSO token.
  • Mapping: Map LDAP attributes (for example mail, uid, cn) to the local users row: create-on-first-login or sync job for role assignment (requester, guide_hod, etc.).
  • Backend: Replace or wrap login in auth.controller.js with an LDAP strategy (e.g. ldapjs or Passport LDAP), returning the same JWT shape as today so the React client does not change.
  • Security: TLS to LDAP; rate limiting on auth; no password logging; optional MFA per institute policy.
  • Fallback: Keep password reset / OTP flows only for supported non-LDAP accounts if the institute requires it.

2. Pre-deployment verification

  • Database: PostgreSQL reachable; backend boot runs ensureSchema (includes fleet_locations).
  • OIC locations: Sign in as OIC → Pickup / drop locations → add at least one active location before requesters can submit the vehicle form (backend enforces active list).
  • SMTP: EMAIL_USER / EMAIL_PASS valid; send a test from staging.
  • Environment: JWT_SECRET, GUIDE_ACTION_SECRET (or reuse JWT secret), FRONTEND_PUBLIC_URL (for guide email links), SUPERVISOR_EMAIL, OIC_EMAIL / APPROVER_EMAIL, ADMIN_EMAIL as needed.
  • Frontend: VITE_API_BASE_URL points to deployed API (/api suffix).
  • CORS: Production origin added in app.js allowedOrigins.
  • HTTPS: Terminate TLS at reverse proxy; cookies / tokens only over secure transport if you migrate to cookies.

3. Session and browser back button

Logout clears sessionStorage. The login form resets on load and when the page is restored from the back-forward cache (pageshow with persisted), and navigation after logout uses replace: true to reduce history stack retention. Institute browsers may still offer saved passwords; that is controlled by browser password manager settings.

4. Email templates

User, guide, supervisor, OIC, and admin notifications use a common Bus Transport Cell layout (plain text and HTML Arial body) with subjects prefixed Sub: where applicable, and the contact block above in footers and allotment mails.

After pulling this release, run the frontend production build (npm run build) and deploy the dist folder behind your web server, with API requests proxied to the Node backend.
